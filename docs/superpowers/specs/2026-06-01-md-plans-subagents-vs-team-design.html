1. Goal & context

This fork added two HTML-emitting behaviours on top of upstream Superpowers: brainstorming emits an HTML design spec, and writing-plans emits a synced HTML plan view plus an HTML multi-session structure. In practice the HTML plan view added little value over the canonical Markdown — it introduced a sync contract (the sp-task-state mirror) that must be regenerated on every checkbox tick, for a document that is already pleasant to read in Markdown.

Two things should change together:

  • Revert the plan HTML. Plans return to canonical Markdown only. The multi-session triage — a valued fork addition — stays, but expressed as Markdown files instead of HTML artifacts.
  • Add an execution-model choice. Before writing tasks, writing-plans asks whether the work will be executed by sequential subagents (today's flow) or a team of specialists working concurrently. The answer changes the shape of the plan document.

The brainstorming HTML design spec (this very document) and the html-artifacts skill are kept — only the plan/roadmap/learnings HTML outputs are reverted.

2. Decision flow

Plan authoring gains one new decision point. Three judgments run in order, and the last two are orthogonal: multi-session decides how many plan documents; the execution model decides the shape of each document.

Scope check → multi-session judgment → execution-model choice. The model choice selects which plan shape each plan document takes.

Composition rule: a multi-session feature produces one roadmap.md and several session plans; each session plan is written in whichever shape the execution-model choice selected. A single-session feature produces one plan document in that shape.

3. The two plan shapes

Both shapes keep the existing quality bar — bite-sized 2–5 minute TDD steps, exact file paths, complete code per step, no placeholders. They differ only in structure and the handoff they point to.

Specialists are inferred per plan

There is no fixed role taxonomy. When writing a team plan, derive the roles the work actually needs from the task content — for example a DB-migration specialist, an API specialist, a React specialist — and tag each task with one. The role is a label that scopes the dispatched agent's instructions; it is not tied to a registered harness agent type.

Team plan header & work-streams A team plan opens with a Work-streams section listing each stream, its specialist, the tasks it owns, and its dependencies on other streams (e.g. "Stream C: API — depends on Stream A: DB schema"). This is the dependency graph a dispatcher reads to decide which streams launch in the same wave.

4. Multi-session in Markdown

The multi-session triage logic is unchanged ("judge the scope first; if it spans subsystems or overflows one lead's context, produce a multi-session structure"). Only its output format changes from HTML artifacts to Markdown files:

docs/superpowers/plans/<feature>/
    roadmap.md            session order, dependencies, status, links to sessions
    session-01-<name>.md  self-contained canonical plan (sequential- or team-shaped)
    session-02-<name>.md
    learnings.md          cross-session memory
  • roadmap.md — orders sessions, shows dependencies and status, links each session plan. Fix/intermediate sessions are inserted here as they arise.
  • Session plans are self-contained — a fresh lead executes a session by reading only that session's plan plus learnings.md. Each is a normal canonical Markdown plan in the chosen shape.
  • learnings.md — after each session, append an entry with four parts: What happened, Deviations (and why), Surprises, Follow-ups (which may trigger roadmap edits or a new session).

No HTML, no sp-task-state mirror. The Markdown checkboxes are canonical and execution skills read and tick them directly.

5. Execution handoff

The end-of-skill handoff branches on the model chosen up front:

Chosen modelHandoff offersSub-skill
SequentialSubagent-driven (recommended) or inline batch executionsubagent-driven-development / executing-plans
TeamDispatch one specialist per independent work-stream in a wave; review; launch the next wave as dependencies cleardispatching-parallel-agents

Because the model is chosen up front and recorded in the plan header, the handoff is deterministic — it does not re-ask.

7. Non-goals

  • No change to brainstorming. It still emits an HTML design spec via html-artifacts.
  • No new fixed role taxonomy. Specialists are inferred per plan.
  • No rewrite of dispatching-parallel-agents beyond what's needed for the team handoff to point at it cleanly.
  • No upstream PR. This is a deliberate fork-local customization.

8. Testing

After the edits, npm run test:html must pass: the remaining validators (findExternalRefs, findUnreplacedMarkers, findSpecDeficiencies) still cover the spec path, and no test references a removed template or helper. The writing-plans change is behavior-shaping skill content, verified by reading the rendered skill rather than by unit tests.
