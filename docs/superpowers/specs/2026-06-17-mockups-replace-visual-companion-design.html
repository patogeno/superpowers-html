Retire the browser-based visual companion as an offered tool and convey visual design through static, self-contained HTML mockups — schematic ones inline in the spec, higher-fidelity product-styled ones spun out to their own files in a per-topic folder.

1. Problem

The brainstorming skill offers a visual companion: a local web server (skills/brainstorming/scripts/server.cjs) that streams mockups to a browser tab and reads back the user's click selections. Upstream v6 invested heavily in it — a per-session key, a sandboxed file server, restart survival — which is real surface area to run, secure, and keep working across platforms.

This fork already makes the human-facing spec a self-contained HTML artifact. Mockups are the same kind of artifact: a human looks at them, nothing machine-parses them. So the companion's live server and interactive-selection machinery is more apparatus than the job needs. We want the visual value — seeing layouts and screens during design — without running a server, and without diverging so hard from upstream that future merges become painful.

2. Goals & non-goals

GoalsNon-goals
Stop offering the visual companion in brainstorming.Deleting server.cjs, the shell scripts, or tests/brainstorm-server/.
Convey visual design as static, self-contained HTML mockups.Any interactive / click-to-select browser viewer.
Inline schematic mockups in the spec; spin out detailed ones to their own files.Changing the canonical stylesheet.css or adding a mockup template.

Fork principle preservedWe unwire the companion rather than delete it. The server, scripts, and their tests stay on disk identical to upstream, so future git merges from upstream don't re-fight the companion. Only the human-read brainstorming guidance changes.

3. Authoring flow

During brainstorming, when a question would be clearer shown than told, the agent authors a mockup instead of opening a server. Whether it lands inline or as its own file depends on whether the narrow spec can hold the detail well.

A visual question yields a mockup; fidelity decides whether it lives inline in the spec or as its own linked file.

Because there is no interactive browser, the user views the mockup by opening the HTML file, and any choice between options is taken in the terminal (normal Q&A) — the same way every other brainstorming decision is made.

4. Fidelity split

The deciding question is not "is this visual?" but "can the spec hold this without becoming a UI file?" Schematic structure belongs inline; pixel-level product fidelity belongs in its own file.

AspectInline schematicSpun-out mockup
Lives indesign.html (the spec)Own file in the topic folder
FidelityStructural — boxes, flow, layout skeletonHigh — represents the real screen
StylingCanonical spec stylesheet (.fig, SVG)The product's own inline CSS
Self-containedrequiredrequired
Spec-deficiency validatorApplies (it is the spec)exempt
Why mockups carry their own styleA spun-out mockup represents the product being designed, so it should look like that product — its own colors, type, and layout, inline. It deliberately does not inline the plugin's spec stylesheet. html-artifacts is reused for its self-containment discipline (one file, zero-dep, no CDNs), not its theme.

5. Output layout

A lone spec stays a single file, exactly as today. The moment a topic produces more than one artifact (a spec plus one or more mockups), everything moves into a dated per-topic folder so the set stays together and is easy to reference.

CasePath
Spec onlydocs/superpowers/specs/YYYY-MM-DD-<topic>-design.html
Spec + mockupsdocs/superpowers/specs/YYYY-MM-DD-<topic>/ containing design.html + mockup-<name>.html

The spec links each spun-out mockup with a plain relative <a href>. When the work proceeds to a plan, the plan links the same files (see §8).
