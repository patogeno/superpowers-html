Make the human-facing collaborative artifacts of Superpowers — the design spec and the implementation plan — render as rich, navigable, self-contained HTML, while leaving the agent-only instruction and execution layer as lean Markdown. Add a multi-session execution model with a cross-session learning loop for large-scope work.

1 · Summary & rationale

Superpowers is a coding-agent methodology built from composable skills. Most skill files are instructions consumed by the model and are best kept in token-cheap Markdown. But a subset of artifacts are produced for a human to read and react to — and there the format matters.

Two articles by Thariq Shihipar (Claude Code team) — "The unreasonable effectiveness of HTML" and "HTML is the new Markdown" — argue that for human-AI collaborative outputs (plans, specs, design explorations), HTML beats Markdown: better information density, navigability, diagrams/tables, and shareability, making the work "legible enough that you actually want to read it." Roughly ~1% of these tokens reach production; the rest exist to make the process legible. This directly matches the project owner's experience: Markdown plans are hard to navigate and dull to engage with, which degrades the quality of feedback during design.

Core insightThe win is at the human review layer, not the agent-instruction layer. We convert what the human reads; we do not bloat what the model executes.

2 · Scope & principles

In scope

  • The design spec produced by brainstorming → self-contained HTML with diagrams and tables.
  • The implementation plan produced by writing-plans → Markdown stays canonical; an HTML view is generated for the human.
  • A multi-session model for large scope: a session roadmap, self-contained session plans, and a running learnings log.
  • A new shared html-artifacts skill holding the canonical stylesheet and HTML authoring conventions.

Principles

  • Additive & mergeable. Make surgical edits to two skills, add new files, and touch nothing agent-only — so upstream improvements from obra/superpowers remain easy to pull.
  • Zero dependencies. A hard project rule. No external libraries, CDNs, or build step. Diagrams are hand-authored inline SVG; CSS is inlined.
  • Self-contained artifacts. Each HTML file stands alone (no external resource references), so it is portable and shareable.
  • Markdown stays authoritative where machines read it. The plan's checkbox task-state must remain machine-parseable and untouched.

Plan view ↔ canonical sync

The agent authors the Markdown plan as the source of truth, then authors/regenerates the HTML view from it in the same turn. Whenever the Markdown changes (including agents ticking checkboxes during execution), the HTML view is regenerated to match. The HTML view mirrors task/step completion state but is never the authority.

5 · Multi-session workflow & learning loop

Trigger: after the spec is approved, writing-plans judges scope. If the work spans multiple subsystems or more tasks than one session can carry while the lead's context stays healthy, it produces the multi-session structure. Otherwise it produces a single plan (rendered as HTML). The trigger is the agent's judgment — no manual knob.

Fig 2 · The learnings log (purple) is the cross-session memory that lets the lead's context reset safely and lets earlier sessions reshape later ones.

Session plan = self-contained

A fresh lead must be able to execute a session by reading only that session plan plus the learnings log — no need to re-read the whole spec or sibling plans. Each session plan therefore restates the minimal context it needs.

Learnings log entry (per session)

  • What happened — what was actually built/changed.
  • Deviations — where execution differed from the plan, and why.
  • Surprises — discoveries, gotchas, constraints found.
  • Follow-ups — new tasks/risks for future sessions (may trigger roadmap edits or a new session).

6 · Styling & the shared html-artifacts skill

To keep every artifact self-contained and avoid CSS drift across skills, a new skill html-artifacts owns the single source of truth:

  • the canonical stylesheet — light/dark variables, typography, the sidebar-TOC plan/spec layout, diagram and table styles (the design system used by this very document);
  • the authoring conventions — write self-contained HTML, inline the CSS, draw diagrams as inline SVG, use semantic tables, never reference external resources.

brainstorming and writing-plans reference it with a **REQUIRED SUB-SKILL** marker (Superpowers' own composition pattern). The agent reads it and inlines the style when producing an artifact. This is purely additive (a new folder), and future-proofs HTML-ifying other human-facing skills later.

7 · Diagrams & tables

Diagrams are hand-authored inline SVG (with CSS box/flow layouts for simpler cases) — chosen over Mermaid-via-CDN and bundled-Mermaid to honor the zero-dependency rule and keep artifacts self-contained and offline-forever. Tables are plain semantic HTML. Both are emphasized in the design/spec phase so architecture and processes are visually graspable. Diagrams use the shared node/edge classes so they theme automatically (every figure in this spec is an example).

8 · Testing & verification

9 · Non-goals (YAGNI)

  • No rewrite of agent-only skills. Behavioral/execution skills stay Markdown; HTML there would only add token noise.
  • No external diagram libraries (Mermaid CDN or bundled) — rejected for zero-dep/self-contained reasons.
  • No build step or converter tooling. The agent authors HTML directly; nothing to compile or maintain.
  • No changes to multi-platform packaging (Codex/Cursor/OpenCode/Gemini) beyond what the additive skills naturally carry.
  • Code-review and finishing-branch menus are not converted in this iteration (see Future work).
